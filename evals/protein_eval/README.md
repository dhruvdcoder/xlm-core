# Protein Structure Evaluation (PAPL-style)

This directory contains everything needed to evaluate the structural quality
of protein sequences generated by a trained model — following the evaluation
protocol from the [PAPL paper (Peng et al., 2025)](https://arxiv.org/pdf/2509.23405).

---

## How it works

The pipeline has three stages:

```
[1] GENERATE          [2] FOLD              [3] MEASURE
your model       →   ESMFold (GPU)    →    aggregate_metrics.py
  (FASTA files)       (PDB files +          (foldability %, diversity,
                       metrics.jsonl)        AA entropy)
```

**Stage 1 — Generate sequences.** A trained masked diffusion model produces
amino-acid sequences by starting from a fully-masked canvas of the target
length and iteratively unmasking tokens. Sequences are written to FASTA files
grouped by length.

**Stage 2 — Fold with ESMFold.** [ESMFold](https://www.science.org/doi/10.1126/science.ade2574)
(Meta AI, ~690M parameters) predicts the 3-D structure of each sequence in a
single GPU forward pass — no multiple-sequence alignment, no physics
simulation. It also outputs three confidence scalars per sequence:

| Metric | Range | Meaning |
|---|---|---|
| **pLDDT** | 0–100 | Per-residue confidence in local backbone geometry. Stored in the PDB B-factor column. |
| **pTM** | 0–1 | Global confidence that the predicted fold resembles a real protein fold, analogous to a TM-score. |
| **pAE** | 0–∞ Å | Average predicted positional error between every pair of residues. Lower = more confident inter-domain geometry. |

**Foldability** (the primary PAPL metric) is the fraction of sequences where
all three thresholds are met simultaneously:

```
foldable = pLDDT > 80  AND  pTM > 0.7  AND  pAE < 10
```

**Stage 3 — Aggregate.** `aggregate_metrics.py` reads the metrics JSONL files
and FASTA files to produce a per-length breakdown matching PAPL Table 1, plus
corpus-level diversity and amino-acid entropy.

## Prerequisites

- A SLURM GPU node — **1× A100-40GB** recommended. A V100-32GB is sufficient
  for sequences up to ~400 amino acids.
- CUDA 12.1 toolkit on `PATH` (needed once for the OpenFold kernel build; the
  matching `cu121` PyTorch wheel is installed automatically).
- Python 3.10 available as `python3.10` (strongly recommended — sidesteps a
  Python 3.11+ bug in `fair-esm`). Check with `python3.10 --version`.
  Python 3.12 also works; `setup_eval_venv.sh` applies the patch automatically.

## One-time setup

Start an interactive GPU session, then run:

```bash
srun --partition=gpu --gres=gpu:1 --mem=40G --cpus-per-task=4 --pty bash
cd /path/to/xlm-core
bash evals/protein_eval/setup_eval_venv.sh
```

This takes 5–15 minutes (mostly compiling the OpenFold CUDA kernel). It
creates a self-contained venv at `vendor/dplm/.venv-esmfold` and prints
`Setup complete` when done. You only ever need to repeat this if you update
PyTorch or switch to a different CUDA toolkit version.

**Using an existing `dplm` clone.** If you already have dplm cloned separately
(e.g. at `/work/.../dplm`), override `DPLM_ROOT`:

```bash
DPLM_ROOT=/work/.../dplm bash evals/protein_eval/setup_eval_venv.sh
```

## FASTA format

Sequences must be organized as follows before running ESMFold:

```
<experiment_dir>/
  by_length/
    iter_<sampling_steps>_L_<length>.fasta   ← one file per length
    iter_<sampling_steps>_L_<length>.fasta
    ...
```

**File naming:** `iter_500_L_200.fasta` means 500 denoising steps, length 200.

**Header format:** `>SEQUENCE_<idx>_L=<length>` — no spaces, no ambiguous
amino-acid characters. Sequences containing `X` are silently skipped by
ESMFold.

**Recommended lengths** to match the PAPL paper: 100, 200, 300, 400, 500
(up to 800aa; see GPU memory note below). Aim for ~100–200 sequences per
length for reliable statistics.

An example with 2 sequences of length 64 is in `example_data/by_length/`.

## Running ESMFold

```bash
FASTA_DIR=evals/protein_eval/<experiment>/by_length \
PDB_DIR=evals/protein_eval/<experiment>/esmfold_pdb \
bash evals/protein_eval/run_esmfold.sh
```

The script activates the eval venv and sets all required environment variables
automatically. Progress is logged to stdout with per-sequence pLDDT / pTM / pAE.

**For sequences ≥ 600aa**, reduce the batch size to avoid GPU OOM:

```bash
MAX_TOKENS=512 FASTA_DIR=... PDB_DIR=... bash evals/protein_eval/run_esmfold.sh
```

**Expected output layout:**

```
<experiment>/esmfold_pdb/
  iter_500_L_200/
    SEQUENCE_0_L=200_plddt_72.3.pdb
    SEQUENCE_1_L=200_plddt_85.1.pdb
    ...
    metrics.jsonl          ← pLDDT, pTM, pAE, foldable per sequence
  iter_500_L_300/
    ...
```

## Aggregating metrics

```bash
python evals/protein_eval/aggregate_metrics.py \
  --fasta-dir evals/protein_eval/<experiment>/by_length \
  --pdb-root  evals/protein_eval/<experiment>/esmfold_pdb
```

**Example output:**

```
════════════════════════════════════════════════════════════════════════
```

Diversity and AA entropy are computed from the FASTA files alone and are
always available, even if ESMFold folding was only partially completed.
Run without `--pdb-root` to get sequence-level stats only.

## Running via SLURM

For a full eval run (hundreds of sequences), submit a batch job:

```bash
#!/usr/bin/env bash
#SBATCH --job-name=esmfold_eval
#SBATCH --partition=gpu
#SBATCH --gres=gpu:1
#SBATCH --mem=40G
#SBATCH --cpus-per-task=4
#SBATCH --time=04:00:00
#SBATCH --output=evals/protein_eval/logs/%x_%j.out

cd /path/to/xlm-core
mkdir -p evals/protein_eval/logs

EXP=my_experiment
FASTA_DIR=evals/protein_eval/${EXP}/by_length
PDB_DIR=evals/protein_eval/${EXP}/esmfold_pdb

FASTA_DIR=${FASTA_DIR} PDB_DIR=${PDB_DIR} \
  bash evals/protein_eval/run_esmfold.sh
```

Typical wall time: ~2–8 hours for 100 sequences × 5 lengths on an A100-40GB,
depending on sequence length.

## Troubleshooting

**`ModuleNotFoundError: attn_core_inplace_cuda`**
The OpenFold CUDA kernel was not compiled. Re-run `setup_eval_venv.sh` on a
GPU node (not a login node).

**`AttributeError: module 'esm' has no attribute 'pretrained'`**
The EvolutionaryScale `esm` package (ESM3) is shadowing Meta's `fair-esm`.
Use the eval venv activated by `run_esmfold.sh` — do not mix it with the
xlm-core training venv.

**`ValueError: mutable default ... use default_factory`**
Python 3.11+ dataclass bug in `fair-esm`. `setup_eval_venv.sh` applies the
patch automatically. If it recurs after a `pip install --upgrade fair-esm`,
re-run `setup_eval_venv.sh`.

For more detail on any of these, see `double-backprop/toy_fasta_eval/README.md`,
which documents the full archaeology of each error.
